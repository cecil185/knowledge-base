## Analysis notes

Key observations from v1 baseline:

1. **False positives cluster around platform/infra content from trusted sources.** The filter over-weights source trust (Databricks, Airbnb Engineering, Fly.io) relative to topic relevance. A trusted source publishing off-goal content should still be dropped.

2. **"Complete guide" vendor SEO pattern is a known problem** — the filter should be stricter about `vendor.com/blog/complete-guide-to-X` patterns from lesser-known vendors.

3. **False negatives are mostly `threshold` cases** where the filter was uncertain but should have been more confident. The LLM is calibrated conservatively, which is correct for auto-ticket but creates noise for keep/drop decisions on the labelled set.

4. **Hypothesis for v2:** Add an explicit gate: "Does the article show evidence of *measurably reducing* human-review burden?" This should cut the infrastructure/platform false positives while preserving true positives on technique-heavy articles.
